MAZEDASH - FULL GAME DESCRIPTION
Current project documentation snapshot
Workspace: repository root
Purpose of this file:
This document describes the current game in detail from a player-facing and system-facing perspective. It is meant as a full overview of what exists in the project right now: progression, mechanics, UI, UX, rewards, bots, challenge systems, cosmetics, audio, ads, and the general flow of the game.

==================================================
1. GAME IDENTITY
==================================================

MazeDash is a neon arcade maze game built in SpriteKit.
The core loop is:
- read the maze quickly
- swipe in a direction
- move through the route efficiently
- collect rewards where relevant
- reach the exit as fast and as cleanly as possible

The game is built around:
- speed
- clean route reading
- commitment
- gimmick learning
- replay for better time / stars
- cosmetic progression

The visual identity is:
- dark futuristic background
- neon cyan / magenta / yellow / green accents
- glossy arcade panels
- compact HUD cards
- animated overlays and feedback effects

The game is not a free-roam maze. It is a high-clarity arcade structure with:
- deterministic movement
- deterministic gimmicks
- deterministic maze generation per mode/seed


==================================================
2. CORE MODES
==================================================

The game currently has these main playable modes:

2.1 Story Mode
- Main structured progression mode
- Uses fixed level indices
- Supports stars, best times, progression unlocks, theme unlocks, cosmetic rewards
- Can also run with an optional bot opponent if bot difficulty is enabled

2.2 Time Challenge
- Endless/chain-style timed mode
- Player completes as many mazes as possible inside a fixed timer
- Available durations:
  - 1 minute
  - 2 minutes
  - 3 minutes
- Tracks best completed maze count per duration
- Uses a dedicated challenge result screen

2.3 Daily Challenge
- One fixed maze per day
- Player chooses bot difficulty:
  - Easy
  - Hard
- Goal:
  - beat the bot
  - earn daily coin reward
  - optionally set a new daily best time
- Daily reward is claim-based per difficulty and date
- Daily completion now uses its own daily-clear result overlay

2.4 Shop
- Cosmetic progression screen
- No real-money purchases
- Uses gameplay-earned coins only
- Includes rewarded ad button for +30 coins with daily limit

Story progression is chapter-based internally.

Current chapter structure:

Chapter 01 - Levels 1 to 10
Name: MAZE BASICS
Mechanics:
- none
Purpose:
- pure maze reading
- movement fundamentals
- speed basics

Chapter 02 - Levels 11 to 20
Name: ONE-WAY SYSTEM
Mechanics:
- one-way only
Purpose:
- commitment
- directional restriction
- route-reading before swiping

Chapter 03 - Levels 21 to 30
Name: BREAKABLE PATHS
Mechanics:
- breakable walls only
Purpose:
- intentional impact choices
- route opening through repeated hits

Chapter 04 - Levels 31 to 40
Name: TELEPORT NETWORK
Mechanics:
- teleporters only
Purpose:
- linked-space reading
- portal-based route logic

Chapter 05 - Levels 41 to 50
Name: SWITCH LOGIC
Mechanics:
- switches / toggle blocks only
Purpose:
- cause and effect
- route state changes

Chapter 06 - Levels 51 to 60
Name: KEY SYSTEM
Mechanics:
- key + locked exit
Purpose:
- the exit cannot be used until the key is collected
- route order matters

Chapter 07 - Levels 61 to 70
Name: FOG ZONE
Mechanics:
- fog only
Purpose:
- memory
- visibility restriction
- navigation under uncertainty

Chapter 08 - Levels 71 to 80
Name: VECTOR SHIFT
Mechanics:
- one-way + teleporters
Purpose:
- route commitment combined with spatial redirection

Chapter 09 - Levels 81 to 90
Name: BREACH CONTROL
Mechanics:
- switch blocks + breakable walls
Purpose:
- mechanical planning

Chapter 10 - Levels 91 to 100
Name: BLACKOUT PORTALS
Mechanics:
- fog + teleporters
Purpose:

Important note:

- So the game currently has chapter-based content structure, but the level browsing UI still uses direct level cards instead of a dedicated chapter screen


==================================================
5. STORY DIFFICULTY SHAPE
==================================================

Each chapter follows the intended pattern:
- levels 1-3 of the chapter:
  - simple introduction
  - low maze complexity
  - learn the gimmick
- levels 4-7:
  - moderate usage
  - more route pressure
- levels 8-10:
  - harder usage
  - denser or more puzzle-like layouts

The design principle is:
- do not randomize difficulty
- do not stack too many mechanics too early
- let the player first learn, then apply, then master


==================================================
6. ACTIVE GIMMICKS / MECHANICS
==================================================

Current mechanics available in the project:
- One-Way / directional pathing
- Breakable Walls
- Switch Blocks
- Keys / Locked Exit
- Teleporters
- Timing Gates
- Fog
- Moving Blocks
- Chaser Enemy

Details for each:

6.1 One-Way
- Tiles force or restrict directional movement
- Creates commitment and route pressure
- Used to make the player think before moving

6.2 Breakable Walls
- Visible cracked blocks
- Must be hit multiple times to break
- Used for deliberate route opening
- The system validates that required breakable usage exists in intended breakable levels

6.3 Switch Blocks
- Floor switch activates/deactivates toggle blocks
- Designed as:
  - floor switch
  - toggle obstacle block
- Used for cause/effect puzzle flow
- Levels are validated to ensure switch usage is actually required

6.4 Keys / Locked Exit
- Current design is not a separate door tile anymore
- Instead:
  - the exit is locked
  - player must collect a key first
  - without key, the exit blocks completion
- The exit shows a lock state when blocked
- When key is collected:
  - lock disappears
  - key follows behind the player visually

6.5 Teleporters
- Paired portal logic
- Move player or bot between linked points
- Used to create spatial redirection and hidden path logic

6.6 Timing Gates
- Periodic open/close logic
- Used mainly in challenge-style content
- Adds timing pressure

6.7 Fog
- Visibility restriction mechanic
- Explored space and visibility are limited
- Used to create orientation and memory pressure

6.8 Moving Blocks
- New mechanic
- Deterministic back-and-forth motion
- Linear path
- Constant speed
- No randomness
- Used for timing and spatial control
- Currently integrated first in challenge generation rather than as a dedicated Story chapter

6.9 Chaser Enemy
- New mechanic
- Enemy follows the player
- Behavior types:
  - direct chaser
  - delayed chaser
- Always slower than the player
- Has delay before reacting
- Used to create pressure without making the game random
- Currently integrated first in challenge generation rather than as a dedicated Story chapter


==================================================
7. MAZE GENERATION AND FAIRNESS
==================================================

Maze generation uses structured configuration per level/mode.
Each level config defines:
- maze size
- loop factor
- branch factor
- orb count
- enabled mechanics
- fog radius
- teleporter pairs
- key count
- switch count
- breakable count
- switch block count
- one-way density
- moving block count
- chaser behavior

Fairness rules already present in the codebase include:
- breakable validation:
  - breakable levels must actually require breakable use
- switch validation:
  - switch levels must contain real switch logic and require switch usage
- challenge generation avoids certain unfair stacks
- chaser speed is kept below player speed
- moving blocks are deterministic, not random

Important design point:
- the game is trying to be hard in a fair way
- not luck-based


==================================================
8. STARS, BEST TIMES, AND PERFORMANCE
==================================================

8.1 Story star system
- Story levels can award 0 to 3 stars
- Best time and max stars are stored per level

8.2 Current star threshold logic
Per level, benchmark data stores:
- easyBotTime
- hardBotTime
- oneStarTime
- twoStarTime
- threeStarTime

Current formula:
- 1 star = easy bot benchmark
- 2 stars = hard + 55% of the gap between easy and hard
- 3 stars = hard * 1.15

This system was already rebalanced to avoid the old problem where 3 stars were too close to a perfect solver benchmark.

8.3 Challenge records
Time Challenge stores:
- best maze count for each duration

8.4 Daily records
Daily Challenge stores:
- best time per day
- reward claim state per difficulty

25. MOVING BLOCK SYSTEM
==================================================

Moving blocks are implemented with these design rules:
- deterministic movement
- linear horizontal or vertical motion
- back-and-forth between two points
- constant speed
- no random behavior

Gameplay purpose:
- timing challenge
- spatial denial
- route waiting pressure

Fairness principles:
- player can read the pattern
- player can wait or route around it
- no hidden randomness


==================================================
26. CHASER ENEMY SYSTEM
==================================================

The chaser system currently supports:
- direct chaser
- delayed chaser

Rules:
- slower than player
- not instant reaction
- has delay before path updates
- must not create unavoidable traps

Failure condition:
- if chaser reaches player, run fails

Important bug already fixed:
- returning to the start tile no longer causes fake death in non-chaser levels


==================================================
27. SWITCH SYSTEM
==================================================

The switch mechanic was recently improved.

Current design intent:
- switch is a floor trigger
- toggle block is a separate obstacle state
- stepping on switch changes toggle blocks

Important rules:
- switch levels must actually require at least one switch usage
- switch and block visuals are separated
- text on blocks was removed so the mechanic reads more like game-space, not debug symbols

Current key system behavior:
- the exit is visibly locked if no key is owned

- the exit lock disappears after the key is collected
- the collected key follows behind the player

Current UI concerns from recent user feedback:
- key readability and lock readability were actively being tuned

Current technical state:
